New feature: Product Distance Tables.

We're going to decouple distance tables from their target groups. Now, distance tables can depend on several target groups!

For example, let's say we have the following target groups.

- EdgeOrientation
R, U, L, D

- EquatorSlice
1: {UFR UFL UBL UBR DFR DFL DBL DBR} {UF UL UB UR DF DL DB DR}
{FR FL BL BR}

- CornerOrientation
{UFR UFL UBL UBR DFR DFL DBL DBR}
1: {UF UL UB UR DF DL DB DR FR FL BL BR}

These target groups have indexes 2048, 7920, and 2187 in the Rubik's Cube group, respectively. However, let's say I want a distance table that tracks the distance from both EdgeOrientation and CornerOrientation being solved! I want to be able to do this while only storing the transition tables for the individual groups. Remember that distance tables are extremely compact (0.25 bytes per state), while transition tables are not (4 bytes per move per state, so 16-72 bytes per state in practice).

I would specify the distance tables I want at the end of the target groups field. For example:

EdgeOrientation * CornerOrientation, EquatorSlice

This would specify a distance table of size 2048 * 2187 = 4478976 that specifies a state's distance from both edge orientation and corner orientation being solved, analogous to a Cartesian product, as well as just a standard distance table of size 7920 of the EquatorSlice. This would take up only slightly more than 1MB.

You can also double-dip. For example:

EdgeOrientation * CornerOrientation, EquatorSlice * CornerOrientation, EdgeOrientation * EquatorSlice

This would specify three distance tables: one of size 2048 * 2187, one of size 7920 * 2187, and one of size 2048 * 7920. This would total about 10 MB in distance tables, while still not needing much space at all for transition tables. 

Note that under this specification, we wouldn't need any distance tables for EdgeOrientation alone (likewise for CornerOrientation alone or EquatorSlice alone). You can still include them if you want, but the most elegant implementation might separate out distance tables into their own class.

All these examples had products of 2 groups, but I would like to support products of 3 or more groups, as long as this doesn't make the implementation extremely complex.

Try to support product groups of up to 16 billion states (which would take up 4 GB, the limit.)

Feel free to choose a different input format if you think it would fit better. Make sure the memory usage stays close to its theoretical minimum. For example, a BFS to fill in the distance table may easily exceed 0.25 bytes per state in its queue, so a slower but more memory efficient O(max_depth * states) algorithm that just sweeps through the entire distance table (max_depth+1) times would be better. 

Pseudocode:
Initialize product distance table to all 3s
Index into product distance table using (((ID 1) * (number of IDs in group 2) + (ID 2)) * (number of IDs in group 3) + (ID 3)) * ...
Initialize the all-identities state to 0. A state is only sliced if all components are solved simultaneiously. The product distance table represents the distance to a state where all components are solved at the same time.
int visited_count = 1
int curr_depth_mod_3 = 0
while (visited_count < total_states) {
    for (table_entry in table) {
        if (table[table_entry] != curr_depth_mod_3) {
            continue
        }
        // at this point, we know that the table entry is either
        // on the frontier, or at least 3 away from the frontier
        // in which case it has no unvisited neighbors.
        for (neighbor in table_entry.neighbors()) {
            if (table[neighbor] == 3) { 
                table[neighbor] = (table[table_entry] + 1) % 3
                visited_count += 1
            }
        }
    }
    curr_depth_mod_3 = (curr_depth_mod_3 + 1) % 3
}

Also, make sure the hot loop stays fast! Finally, all else being equal, prefer a simpler implementation, since we're going to add stuff on top of this.
